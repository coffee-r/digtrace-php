# audience: requirements

統合仕様候補を、追加開発や要件定義担当向けに変換してください。

重視すること:

- 現行仕様候補
- 入力条件と結果
- 例外・拒否条件
- 副作用
- 未確認事項
- 追加ヒアリング項目

「仕様として断定可」という表現は避け、次の区分で書いてください。

- **観測で確認**: endpoint、pattern、count、status、effects、response shape などに直接あること
- **仕様候補**: 観測事実と実装事実から要件候補として扱えること
- **推論**: 業務理由、条件名、入力条件の意味づけ
- **未確認**: 追加観測、コード確認、ヒアリングが必要なこと

入力条件、レスポンス形式、セッション/認証由来の値は、根拠がある場合だけ断定してください。count=1 は必ず低観測として明記してください。
